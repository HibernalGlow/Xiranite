## 核心原则：谁拥有生命周期


**前端不该直接拉起后端**，原因：

1. WebView 没有可靠的进程管理（崩溃、退出、权限、隐藏窗口）。
2. 网页端根本没有 spawn 能力，会立刻分叉两套架构。
3. 安全边界会糊掉（前端一旦能起任意进程就危险）。

正确做法：**宿主拥有后端；前端只拿通道。**

## 更好的通信方式（按推荐顺序）

### 方案 A（最推荐）：宿主代理 + 固定逻辑入口，端口对前端不可见

桌面里让 Wails/Tauri **自己当 reverse proxy / asset server**：

```text
前端请求：
  /xiranite-api/...     → 宿主转发到本地 backend
  /reader/...           → 宿主转发到 NeoView controller
  /local-files?...      → 宿主转发

前端永远只打：
  relative URL（同源）
  或  window.location.origin + '/xiranite-api'
```

好处：

- **前端零端口概念**，没有 random port 焦虑
- 后端仍可用 `127.0.0.1:随机端口` 做内部监听，**只对宿主可见**
- 网页模式继续用真正的 HTTP baseUrl（manifest / 固定端口）
- 换 Tauri 时换「宿主代理实现」，前端协议不变

你们已经有一半基础：生产态 Wails 注入 config、dev 写 manifest。差的是把「前端必须知道 `http://127.0.0.1:随机端口`」改成「前端永远打同源路径」。

### 方案 B：固定端口 + 单实例锁（简单，但有坑）

例如永远 `127.0.0.1:18765`（或配置文件写死）：

```text

  try bind 18765
  if occupied:
    检查是否是自己的健康实例 → 复用
    否则报错 / 提示占用，不要静默换端口
```

适合「本机单开」产品。  
不适合多 agent 并行开发（你们 `docs/development-sessions.md` 已经为并发会话设计了 per-port manifest）。

**原则：要么固定且拒绝抢端口，要么随机但前端永远不感知。最讨厌的是「半随机半发现」。**

### 方案 C：命名管道 / Unix Domain Socket / 抽象命名空间

Windows 可用 named pipe：`\\.\pipe\xiranite-backend-<session>`  
Linux/macOS：UDS `/tmp/xiranite-....sock`

- 不占 TCP 端口，更干净
- 但浏览器 `fetch` **不能直接打 named pipe**
- 所以桌面仍要：Host bridge 或 Host 把 pipe 转成 HTTP

这是「后端进程间」的好方案，不是「给 WebView 直接用」的方案。

### 方案 D：纯 Wails/Tauri RPC（Bindings / invoke）

Wails 已有 `Call.ByName('main.XiraniteService....')`，Tauri 有 `invoke`。

适合：

- 窗口控制、文件系统、剪贴板、对话框、重启后端
- 小 payload 的控制面

**不适合**作为 Reader 主通道：

- 大图流、缩略图、Range、缓存、并发下载，HTTP 更合适
- 网页端没有同等 bindings

所以正确分层是：

```text
Control plane  → Host IPC（重启后端、取配置、窗口、对话框）
Data plane     → HTTP/WS（Reader 资源、缩略图、会话 API）
```

你们现在其实已经在走这个方向，只是 data plane 还把「随机端口」暴露给了前端。

## 上 Host Proxy 后，会怎样？

分三种情况：

### 1. 只代理 API，不管图片 URL（半吊子）

- `fetch('/reader/sessions')` 走宿主同源反代  
- 但 session 返回的 `assetUrl` 仍是 `http://127.0.0.1:随机端口/reader/.../page/...`

**结果：** 图片通常仍能显示（WebView 能打本机端口），但你还是暴露了随机端口，**随机端口问题没真正消掉**。  
而且后端重启换端口后，旧 `assetUrl` 会整批 404——这才是最讨厌的 stale URL。

### 2. 正确做法：代理路径 + 改写 baseUrl（推荐）

宿主反代 `/reader/*`，后端生成 URL 时用**前端可见的 base**：

| 环境 | `baseUrl` 应是什么 |
|------|-------------------|
| Wails/Tauri 桌面 | `https://wails.localhost` / `http://tauri.localhost` / `window.location.origin` |
| 开发 Vite | `http://127.0.0.1:5173`（由 Vite proxy 转到后端） |
| 纯网页远程 | 真实 API 域名 |

于是 `assetUrl` 变成：

```text
http://127.0.0.1:5173/reader/s/.../page/...?token=...
// 或桌面
https://wails.localhost/reader/s/.../page/...?token=...
```

`<img src>` 仍然是 **HTTP 直通语义**，只是打到**稳定入口**，不再打随机端口。  
**图片协议不用换，换的是入口。**

### 3. 错误做法：想用纯 IPC 传图片字节

Wails Call / Tauri invoke 拉 base64 再塞 `blob:`：

- 大图、缩略图条、预加载、Range、缓存、取消都会变难  
- 网页端也不通用  

**不建议**把主图通道改成 IPC。

## 真正要注意的影响点（比“能不能显示”更关键）

### 1. `<img>` 不会带自定义 Header

你们 token 放在 query（`?token=`）是对的，因为：

```text
fetch() 可以加 Authorization header
<img src> / CSS background / video 通常不能方便地带自定义 header
```

Host proxy **必须继续支持 query token**（或 cookie 会话），不能只认 header，否则图片全挂。

### 2. 必须透传二进制响应语义

反代不能当普通 JSON 网关随便改 body。需要原样透传：

- `Content-Type`（jpeg/png/webp…）
- `Content-Length` / `Transfer-Encoding`
- `Accept-Ranges` / `Content-Range`（视频/大文件 seek）
- `ETag` / `Cache-Control` / `Last-Modified`
- 状态码 `200` / `206` / `304` / `404`

否则会出现：JSON API 正常，但大图裂、视频 seek 坏、缩略图缓存失效。

### 3. 跨源 vs 同源

现在前端页面在 `5173`，图片在 `随机后端端口`，是 **跨 origin 图片请求**（多数情况 `<img>` 能加载，但 canvas 读像素会 taint）。

改成同源 proxy 后：

- 更干净
- canvas / 滤镜 / 取像素更不容易踩 CORS
- cookie 方案才可行

### 4. 后端重启后的 URL 寿命

若 `assetUrl` 写死随机端口：

```text
后端重启 → 端口变 → 旧 img src 全死
```

若 `assetUrl` 始终指向稳定 origin：

```text
后端重启 → host 重连内部端口 → 同一 URL 继续可用
```

这正是 host-proxy 对图片链路的**最大收益**。

### 5. 网页端远程查看

远程网页本来就该是：

```text
https://your-host/reader/s/.../page/...?token=...
```

和桌面 host-proxy 形态一致，只是没有“本机随机端口”这一层。  
**图片 HTTP 直通在网页端是正确且必要的。**

```text
               │  仍然要有一条内部通道
               ▼
┌─────────────────────────────┐
│  Bun/Node backend           │
│  监听 127.0.0.1:内部端口     │  ← 这里常常还是端口
└─────────────────────────────┘
```

| 段 | 是不是端口 | 谁关心 |
|----|------------|--------|
| 前端 → 宿主 | 通常**不是**业务后端端口（同源 `/reader`，或 Wails/Tauri 内置资源服务） | 前端 |
| 宿主 → Bun/Node | **多数情况下是** `127.0.0.1:随机或固定端口` | 只有宿主 |

所以：

- **对前端：不再是“端口通信问题”**
- **对宿主内部：多半仍是 loopback TCP**

---

## “反代”具体在干什么

反代不是魔法协议，只是：

```text
前端请求：  GET /reader/s/xxx/page/1
宿主收到后：GET http://127.0.0.1:53127/reader/s/xxx/page/1
            └─────────────┬──────────────┘
                          内部 baseUrl，前端永远看不到
```

Bun/Node 还是那个 HTTP server：

```ts
server.listen(0, "127.0.0.1")  // 或固定端口
```

Go/Rust 只是当“知道内部地址的那个人”，把请求转发过去，把响应原样吐回 WebView。

---

## 内部通道不一定非要 TCP 端口

宿主 ↔ Bun 还可以是：

| 方式 | 还是不是“端口” | 说明 |
|------|----------------|------|
| `127.0.0.1:随机端口` | 是 TCP 端口 | 最常见、最省事，和现有 Bun HTTP 几乎零改 |
| 固定端口 `18765` | 是 | 简单，但多开会撞 |
| Unix Domain Socket | 不是 TCP 端口 | Linux/mac 很合适；Windows 也有路径形态 UDS（视运行时支持） |
| Named Pipe `\\.\pipe\...` | 不是 TCP 端口 | Windows 原生；Bun/Node 要额外适配，浏览器仍打不了，必须宿主转 |
| stdio / 自定义 RPC | 不是端口 | 适合控制命令，不适合大图流式 HTTP |
| 内嵌运行时（Go 里直接跑 JS？） | 进程内 | 现实成本高，一般不这么干 |

**实务上你们最划算的仍是：**

> 内部继续 `127.0.0.1 + 端口`（甚至继续 random）  
> 外部（前端）永远不碰这个端口

随机端口的讨厌之处，来自**暴露给前端 + 发现/失效**；藏在宿主里就不可恨了。

---

## 为什么内部还用端口通常没问题

1. **只绑 loopback**：`127.0.0.1`，外网进不来  
2. **只有宿主知道**：前端、书签、日志里都看不到  
3. **和现有栈匹配**：你们 backend 已是 HTTP + `assetUrl` + Range + token  
4. **图片继续 HTTP 语义**：宿主转发的还是 HTTP 响应，`<img>` 无感  

若强行上 named pipe 却仍要服务 `<img src>`，最后还得在宿主侧再做成 HTTP 入口——等于多绕一圈，收益不大。

---

## 一句话

> **反代之后，前端不再跟 Bun 做端口通信；是宿主跟 Bun 做内部通信，而这条内部链路在默认方案里仍然常常是本机 TCP 端口。**  
> 端口没有消失，只是从「产品契约」降级成了「宿主实现细节」。

如果你关心的是「能不能彻底消灭端口」：可以，但那是 UDS/pipe/内嵌，属于优化项；**不是解决当前痛点的必要条件**。当前痛点用「内部端口 + 对外同源反代」就够了。

这些是验收成本，不是架构重写成本。

## 两个端口，别混在一起

| 端口 | 是什么 | 开发要不要 | 生产桌面要不要 |
|------|--------|------------|----------------|

### 现在 / 做完 P0 之后（推荐 dev 形态）


|----------|------|----------|----------|

## A. 前端自己当反代（改动最小）

Vite 固定端口 + proxy：

```ts
// 概念配置
server: {
  host: "127.0.0.1",
  port: 5173,
  strictPort: true, // 被占用就失败，绝不偷偷换端口
  proxy: {
    "/reader": "http://127.0.0.1:内部后端",
    "/local-files": "http://127.0.0.1:内部后端",
    // 其它 API 前缀
  },
}
```

你只记：

```text
http://127.0.0.1:5173
```

- 浏览器打开它  
- Wails dev 也加载它  
- 图片 `assetUrl` 也是 `http://127.0.0.1:5173/reader/...`  

**这就是「前端也反代」。**  
后端端口前端完全看不见。

代价：

- 5173 被别的进程占用时要处理（`strictPort` 直接报错，或你们文档里的多会话换端口）
- 多 agent 并行时，不可能所有人共用同一个 5173——要么一人一端口，要么一人一网关端口

## B. 单独固定网关端口（更像「一个产品网址」）

supervisor / Go 监听**永远同一个**端口，例如 `9100`：

```text
http://127.0.0.1:9100          ← 你唯一要记的
  → 反代页面到 Vite
  → 反代 API/图片到 Bun
  → HMR WebSocket 也转发
```

好处：

- 书签、文档、Wails `FRONTEND_DEVSERVER_URL` 全写死一个
- 以后换 Vite 端口、换后端端口，对外网址不变
- 和「生产由宿主统一入口」心智一致

代价：

- 多一层进程/转发
- HMR websocket 要正确代理（可做，不是黑科技）
- 多会话时：每人一个网关端口，或单实例锁「只能开一份 dev」

## 多 agent / 多开怎么处理

你们现在为并行开发设计了「前端端口可变」。固定网址和多开有张力：

| 策略 | 含义 |
|------|------|
| **单实例固定网址** | 永远 `:9100`，第二开拒绝或接管 | 最符合「一个固定网址」 |
| **主固定 + 并行可覆盖** | 默认 `:9100`；`XIRANITE_DEV_URL=...` 才换 | 日常省心，并行仍可 |
| **按会话派生** | `:9100`、`:9101`… 但仍是「每会话一个固定入口」 | 保留并行，接受多个固定号 |

建议产品默认：**单实例固定入口**；并行开发用环境变量显式开第二会话。
